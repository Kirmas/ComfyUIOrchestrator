# ТЗ: Оркестратор генеративних пайплайнів над ComfyUI

## 1. Загальна ідея

Веб-застосунок (окремий UI + backend сервер) поверх одного або кількох інстансів ComfyUI та/або платних image-API. Юзер (або агент через API/MCP у майбутньому) будує творчий пайплайн з кроків: text-to-image → вибір найкращого варіанта → multiview → ... → 3D-модель. Система модульна: кожен крок — це "нода-шаблон", яка виконується на будь-якому доступному бекенді, що має відповідну capability.

Ключові властивості:
- Кілька інстансів ComfyUI, балансування навантаження між ними; якщо жоден не вільний — джоба чекає в черзі (в т.ч. чекає запуску хоча б одного інстансу).
- У майбутньому — платні API-провайдери (напр. Nano Banana / Gemini image, Fal, Replicate) як альтернативний бекенд для тих самих нод.
- Кожна нода генерує N варіантів, юзер відмічає найкращі, наступні кроки беруть відмічені виходи як вхід.

## 2. Архітектура

### 2.1 Компоненти
- **Backend**: Python, FastAPI. Причина: ComfyUI сам на Python, простіше інтегруватись.
- **Черга**: Redis + Arq (легша альтернатива Celery). Джоби генерації йдуть у чергу, воркер-диспетчер роздає їх бекендам.
- **БД**: PostgreSQL (метадані: бекенди, capability, кроки, треки, джоби, асети).
- **Сховище асетів**: S3-сумісне (MinIO). Картинки, multiview-сети, 3D-файли. У БД — тільки посилання.
- **Frontend**: React. Кастомна сітка (CSS Grid) + SVG-оверлей для стрілок. НЕ react-flow / не повний graph-editor (див. розділ 5).
- **Realtime**: WebSocket наскрізь — ComfyUI віддає прогрес по WS, бекенд проксує це на фронт (статус джоб, прогрес генерації, поява результатів).

### 2.2 Абстракція Backend Provider
Єдиний інтерфейс для всіх виконавців:

```python
class JobBackend(Protocol):
    async def submit(self, execution_config: dict, inputs: dict) -> str  # job_id
    async def status(self, job_id: str) -> JobStatus  # pending|running|done|error
    async def result(self, job_id: str) -> list[AssetRef]
    async def capacity(self) -> CapacityInfo  # queue_length, is_alive
```

Реалізації:
- **ComfyUIBackend** — обгортка над HTTP/WS API конкретного інстансу (`POST /prompt`, `GET /history/{id}`, WS `/ws` для прогресу). Heartbeat через `/system_stats` або `/queue` кожні N секунд для реєстру живих інстансів.
- **ApiBackend** — обгортка над платним API. Перед використанням перевіряє: (а) юзер має API-ключ для провайдера, (б) юзер явно дозволив використання цього провайдера. Для диспетчера виглядає як бекенд з майже необмеженою capacity.

Оркестратор і UI не знають різниці між типами бекендів.

### 2.3 Capability Mapping (центральна ідея конфігурації)
Три сутності:
1. **NodeType** — абстрактний тип кроку (`text_to_image`, `multiview`, `image_to_3d`, ...). Те, що бачить юзер.
2. **Backend** — конкретний інстанс ComfyUI або API-провайдер.
3. **Capability** — зв'язка "цей бекенд вміє цей NodeType ось так":

```
capability {
  backend_id
  node_type_slug
  enabled: bool
  execution_type: "comfyui_workflow" | "api_call"
  config: {
    -- comfyui_workflow:
    workflow_json,        -- експортований API-формат workflow ComfyUI
    param_mapping,        -- поля ноди-шаблона -> input-и workflow
    -- api_call:
    provider,             -- "nano_banana" | ...
    model_id,
    requires_api_key: true,
    param_mapping
  }
}
```

Приклад: сервер A має multiview через workflow_X.json, сервер B взагалі не має цієї capability (диспетчер його ігнорує для таких нод), сервер C — інший workflow з іншою моделлю, API-бекенд — прямий виклик провайдера.

Додавання нового сервера/провайдера/типу ноди = записи в БД, без змін коду диспетчера і UI.

### 2.4 Диспетчеризація джоби
1. Джоба ноди `X` стає в чергу.
2. Диспетчер бере всі backend з `capability.enabled=true` для цього node_type.
3. Фільтрує по правах юзера (для api_call: є ключ + явний дозвіл юзера на використання платного API для цієї ноди).
4. Враховує вибір юзера в UI ноди: "Auto (balance)" / тільки ComfyUI / тільки API / конкретний сервер вручну.
5. З решти бере найменш завантажений. Якщо нікого нема — джоба чекає в черзі (Redis це дає з коробки), фронт бачить статус "waiting for backend".
6. Якщо юзер просить N варіантів — система може розкидати N підджоб по різних інстансах паралельно.

## 3. Ноди: generic template engine (НЕ хардкод типів)

- Одна універсальна нода з **param_schema** (JSON Schema-подібна): типи полів — `image`, `text`, `int`, `float`, `seed`, `enum`, `file`. Кількість input-слотів і output-ів описується в шаблоні.
- Фронт має один generic-рендерер форм по схемі.
- Для окремих node_type можна реєструвати **кастомний renderer plugin** (напр. сітка превʼю 2x2 для multiview, 3D-вʼювер для image_to_3d). Якщо кастомного нема — fallback на generic. У MVP достатньо generic + максимум 1-2 кастомних.
- Шаблони нод конфігуруються в налаштуваннях (адмінка/settings): назва, схема полів, дефолти, до якого node_type_slug належить.

## 4. Модель даних пайплайна: дерево генерацій + Track/Swimlane

НЕ довільний граф. Структура:

```
Track (лінія/доріжка) {
  id
  project_id
  row_index               -- позиція рядка в сітці (статична в MVP)
  spawned_from: {node_id, output_id} | null   -- null = стартова або "нова підзадача після розриву"
}

Node (клітинка/крок) {
  id
  track_id
  step_index              -- колонка, зліва направо
  template_id
  inputs: [InputRef]      -- див. нижче
  params: {...}           -- значення полів по схемі шаблона
  outputs: [{asset_id, selected: bool}]
  status: draft|queued|running|done|error|discarded
  backend_used
  requested_variants: N   -- скільки варіантів згенерувати
}

InputRef =
  | {type: "self_prev"}                -- дефолт: попередня клітинка своєї лінії (selected output)
  | {type: "track_below_prev"}         -- дефолт для другого інпута: сусідня лінія знизу
  | {type: "explicit", node_id, output_id}  -- юзер вручну вказав будь-яку клітинку
  | {type: "upload", asset_id}         -- юзер завантажив свою картинку
  | {type: "text", value}             -- текстовий промпт як стартовий вхід
```

Правила семантики (зафіксовані з обговорення):
- **Fan-out**: з одного output можна породити кілька нових Track (крупний план голови / multiview / розбиття на частини — три лінії від однієї картинки). Кожен обраний чекбоксом output теж може породити окрему лінію.
- **Multi-input**: дефолт = своя лінія + одна знизу; юзер може перетицяти другий інпут на будь-яку іншу клітинку.
- **Джерело не "з'їдається"**: трек, з якого взяли output для merge/spawn, продовжує жити далі незалежно (він теж може бути частиною fan-out).
- **Тупикова лінія + розрив**: юзер може завершити лінію; після порожньої клітинки в тому ж рядку може початися нова непов'язана підзадача (`spawned_from=null`). Візуально розрив показується явно (gap + пунктир/іконка "нова гілка"), щоб було зрозуміло, що це не продовження.
- **Discard/re-roll**: юзеру не сподобались усі N варіантів → крок позначається discarded, створюється новий крок з тими ж inputs (нова генерація N варіантів).
- **Branching**: модель — дерево, не список. Юзер може повернутись до старого кроку, вибрати інший output і піти іншою гілкою.

## 5. UI

- Сітка: колонки = кроки (зліва направо), рядки = Track-лінії (зверху вниз). CSS Grid.
- Звичайне продовження лінії НЕ малюється стрілкою — сусідні клітинки і так зрозумілі.
- SVG-оверлей малює стрілки тільки для: (а) merge (нода бере вхід не зі своєї лінії), (б) spawn (звідки народилась нова лінія). Якщо треки не сусідні — лінія проходить через проміжні рядки (як у git-графах).
- Клітинка-нода: зліва input-слот(и), центр — параметри (generic форма по схемі), справа — сітка з N output-варіантів з чекбоксами.
- Стартова клітинка: юзер або завантажує картинку, або вводить текст, або одразу ставить сконфігуровану темплейт-ноду.
- Вибір бекенда в ноді: dropdown "Auto / конкретний сервер / тільки API" (фільтр по capability-таблиці).
- **Auto-reflow рядків (перестановка треків, щоб залежні були поруч) — НЕ в MVP.** Друга ітерація. Якщо знадобиться — barycenter-евристика або elkjs/d3-dag. У MVP позиції рядків статичні.

## 6. Що НЕ входить в MVP (свідомо відкладено)
- Auto-reflow / автоматичне перекладання рядків.
- MCP-сервер для агентів. Робиться пізніше як тонка обгортка над REST API (tools: `create_generation_step`, `get_step_status`, `select_output`, ...). REST API з самого початку проєктувати чистим, не привʼязаним до фронта — щоб MCP-шар потім став ~1 днем роботи.
- Платні API-провайдери можна зробити другою чергою після ComfyUI-адаптера, але інтерфейс JobBackend і поля capability під них закласти одразу.
- Розробка самих ComfyUI workflow-ів (text2img, multiview, 3D) — окрема робота поза кодом оркестратора; система приймає готові workflow JSON (API format).

## 7. Порядок робіт (орієнтир)
1. БД-схема + FastAPI скелет (backends, capabilities, node_templates, projects, tracks, nodes, jobs, assets, users/api_keys).
2. ComfyUIBackend-адаптер: submit workflow, WS-прогрес, забір результатів у MinIO. Heartbeat/реєстр інстансів.
3. Черга (Arq) + диспетчер з capability-фільтрацією і балансуванням.
4. Generic template engine (схеми полів, param_mapping у workflow).
5. Frontend: сітка Track/Node, стартова клітинка (upload/text), генерація N варіантів, чекбокси, наступний крок, discard/re-roll, SVG-стрілки merge/spawn, WS-статуси.
6. ApiBackend (один провайдер для прикладу) + перевірка ключів/дозволів.
7. Поліш, помилки, ретраї джоб.

Орієнтовна оцінка: ~3 тижні на MVP (один розробник + Claude Code, з код-ревʼю).

## 8. Відкриті питання (уточнити в замовника перед/під час розробки)
1. **Auth**: один локальний юзер (self-hosted, без реєстрації) чи мультиюзер з акаунтами? Впливає на users/api_keys/permissions.
2. **Деплой**: docker-compose (FastAPI + Redis + Postgres + MinIO) як цільовий формат? ComfyUI-інстанси — зовнішні, підключаються по URL?
3. **Формат param_mapping для ComfyUI**: мапити по id ноди у workflow JSON чи по title ноди? (title стійкіший до перезбирання workflow, id простіший).
4. **Ліміти**: максимум N варіантів за раз? Таймаут джоби? Політика ретраїв при падінні інстансу посеред генерації (перекинути джобу на інший інстанс?).
5. **Скасування**: чи потрібна кнопка "cancel" для запущеної джоби (ComfyUI має /interrupt, але для чужих джоб у черзі інстансу — /queue delete)?
6. **Зберігання**: чи чистити discarded-варіанти з MinIO автоматично (retention), чи зберігати все?
7. **3D-превʼю**: який формат на виході (glb/obj)? Вбудований вʼювер (three.js) потрібен у MVP чи досить download-лінка?
8. **Права на API-провайдери**: дозвіл на рівні "провайдер" чи на рівні "провайдер + node_type" (як обговорювали — краще друге)?
